**BandLab의 장점을 흡수한, "가장 직관적인" 입문자용 모바일 DAW를 AI 코딩 도구와 함께 만드는 단계별 프롬프트 설계서**

## 0. 이 문서 사용법

### 진행 규칙 3가지

1. **순서를 지킨다.** 오디오 엔진(Phase 1)이 화려한 UI보다 먼저입니다. DAW는 "소리가 정확한 타이밍에 나는 것"이 전부이며, 이 토대 없이 UI부터 만들면 나중에 전부 다시 짜게 됩니다.
2. **한 번에 한 Phase.** 프롬프트 붙여넣기 → 결과 확인 → 아래 "완료 기준" 전부 체크 → `git commit` → 다음 Phase. 여러 Phase를 한 프롬프트에 욱여넣지 마세요.
3. **완료 기준 미달이면 넘어가지 않는다.** 부록 B의 디버깅 프롬프트로 고친 뒤 진행합니다.

### 준비물

- Node.js 20 이상, git
- AI 코딩 도구 1개 (Claude Code / Cursor 권장 — 파일 시스템 접근이 자유로워 이 플레이북과 궁합이 좋음)
- 테스트용 스마트폰 (개발 PC와 같은 와이파이에 연결, `npm run dev -- --host`로 접속)
- **유선 이어폰** (블루투스는 자체 지연이 100ms 이상이라 개발 중 오디오 타이밍 판단을 망칩니다)

### 예상 일정 (참고)

|---|---|---|
| 1주차 | 0 ~ 2 | 소리 나는 뼈대 + 타임라인 |

| 3주차 | 6 ~ 8 | 믹서 + 이펙트 + 저장/내보내기 |

## 1. 제품 정의 — 북극성 문서

프롬프트를 던지기 전에 "무엇을 만드는가"를 먼저 고정합니다. 아래 내용은 마스터 프롬프트(3장)에 그대로 녹아 들어갑니다.

### 1.1 한 줄 비전

> **"음악을 전혀 몰라도, 앱을 연 지 30초 안에 첫 소리를 재생하고 3분 안에 8마디 비트를 완성하는 모바일 DAW"**

### 1.2 타깃 사용자

- **주 타깃:** 음악 제작 경험 0~1년의 입문자. DAW라는 단어 자체를 모를 수 있음
- **부 타깃:** 이동 중 아이디어 스케치가 필요한 중급 프로듀서 (데스크톱 DAW로 이어가는 워크플로)

### 1.3 핵심 UX 원칙 8가지 — 모든 프롬프트에 반복 주입되는 가치

1. **30초 규칙:** 첫 실행 → 첫 소리 재생까지 30초 이내
2. **3탭 규칙:** 핵심 동작(재생, 루프 추가, 녹음 시작)은 어떤 화면에서든 3탭 이내
3. **빈 화면 금지:** 새 프로젝트도 스타터 루프와 함께 시작. 흰 도화지는 입문자에게 공포임
4. **즉각적 청각 피드백:** 모든 탭은 소리 또는 햅틱으로 응답. 소리 앱에서 무음 반응은 버그로 취급
5. **실수 안전망:** 모든 편집은 실행취소 가능(20단계). 삭제 등 파괴적 동작은 확인 요청
6. **점진적 노출:** 첫 화면의 컨트롤은 5개 이하. 고급 기능은 항상 "한 겹 아래"에
7. **엄지 존:** 핵심 컨트롤은 화면 하단 1/3에 배치 (한 손 조작 전제)
8. **48px 규칙:** 모든 터치 타깃 최소 48×48px

### 1.4 BandLab에서 가져올 것 / 다르게 할 것

| 구분 | 내용 |
|---|---|
| **가져올 UX 패턴** | 무료 즉시 시작(로그인 강요 없음) · 루프 기반의 빠른 첫 성공 경험 · 원탭 이펙트 프리셋 · 프로젝트 BPM에 자동 동기화되는 루프 미리듣기 |
| **더 쉽게 바꿀 것** | 첫 화면 정보량 대폭 축소 · "장르 선택 → 스타터 프로젝트 자동 생성" 온보딩 · 전문용어 제거(예: Quantize → "박자 맞추기") · 키/스케일 잠금 키보드로 "틀린 음이 없는" 연주 경험 |
| **v1에서 뺄 것** | 소셜 피드, 클라우드 협업, 마스터링 서비스 (전부 v2로) |

> ⚠️ **법적 주의:** BandLab은 UX 패턴의 벤치마크일 뿐입니다. 화면 디자인·아이콘·명칭·사운드 에셋을 모방하지 말고, 브랜딩과 비주얼은 독자적으로 설계하세요. 번들 루프는 반드시 CC0 등 상업적 사용이 허가된 소스만 사용합니다.

### 1.5 MVP 범위 (스코프 고정 — 흔들리면 완성 못 합니다)

**포함:** 최대 8트랙 멀티트랙 타임라인 · 번들 루프 40~60개 · 드럼패드(4×4) · 스케일 잠금 키보드 · 마이크 녹음 · 트랙별 볼륨/팬/뮤트/솔로 믹서 · 원노브 이펙트 3종 · 로컬 저장/자동저장 · WAV 내보내기 + 공유 · 온보딩

**제외(v2):** 클라우드 동기화 · 협업 · 소셜 · AI 마스터링/스템 분리 · MIDI 컨트롤러 · 서드파티 플러그인 · 오디오 타임스트레칭(고품질)

## 2. 기술 스택 결정

### 권장 경로: **웹 우선(PWA) → 필요 시 Capacitor로 앱 패키징**

| 레이어 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | React 18 + TypeScript + Vite | AI 코딩 도구가 가장 정확하게 다루는 스택 → 반복 속도 최상 |
| 오디오 엔진 | **Tone.js** | Transport(마스터 클록)·신스·샘플러·이펙트가 내장. 오디오 엔진을 밑바닥부터 짤 필요가 없어 바이브코딩 성공률이 극적으로 올라감 |
| 상태 관리 | Zustand | 보일러플레이트가 적어 AI가 실수할 표면적이 작음 |
| 스타일 | Tailwind CSS | 디자인 토큰을 프롬프트로 강제하기 쉬움 |
| 저장 | IndexedDB (`idb` 라이브러리) | 오디오 blob + 프로젝트 JSON 로컬 저장 |
| 패키징 | Capacitor (Phase 10, 선택) | 웹 코드 그대로 iOS/Android 앱으로 |

### 대안 경로 비교

| 경로 | 장점 | 단점 | 바이브코딩 적합도 |
|---|---|---|---|
| **웹 PWA + Tone.js (권장)** | 반복 최속, 폰에서 즉시 테스트, 엔진 내장 | 네이티브 대비 지연 ~20-50ms | ★★★★★ |
| React Native + Expo | 네이티브 UX | 오디오 라이브러리 제약 심함, 지연 제어 어려움 | ★★★ |
| Flutter | 성능 균형 | 오디오 생태계 얇음, AI 도구 정확도 중간 | ★★★ |
| 네이티브 (Swift/AudioKit, Kotlin/Oboe) | 최저 지연, 최고 품질 | 전문 지식 필수, 바이브코딩으로 완주 어려움 | ★ |

> **한계를 알고 시작하기:** 웹 오디오는 실시간 기타 모니터링 같은 초저지연 용도에는 부족하지만, **루프 배치 + 패드 연주 + 보컬 녹음** 중심의 입문자용 DAW에는 충분합니다. 지연이 병목이 되는 시점(v2)에 네이티브 오디오 모듈을 검토하세요.

## 제품 요약

## 기술 스택 (임의 변경 금지)
- React 18 + TypeScript + Vite, PWA
- 오디오: Tone.js (package.json에 버전 고정). 모든 오디오는 Tone을 경유하며
  AudioContext를 직접 생성하지 않는다.
- 상태: Zustand / 스타일: Tailwind CSS / 저장: IndexedDB(idb)
- 지원 대상: iOS Safari 16+, Android Chrome 최신. 데스크톱은 개발 편의용.
- 새 라이브러리 추가 전에는 반드시 나에게 먼저 물어본다.

## 오디오 절대 규칙
1. Tone.start()는 반드시 사용자 제스처(탭) 핸들러의 동기 흐름 안에서만 호출.
2. 모든 사운드 스케줄링은 Tone.js의 Transport 기준. setTimeout/setInterval로
   오디오 타이밍을 제어하지 않는다.
3. 재생헤드 등 재생 중 UI 갱신은 requestAnimationFrame으로. 오디오 틱마다
   React state를 갱신하지 않는다.
4. 사용이 끝난 오디오 노드는 dispose()로 해제한다 (메모리 누수 금지).
5. visibilitychange로 백그라운드 복귀를 감지해 컨텍스트 상태를 점검·복구한다.

## UX 절대 규칙
1. 터치 타깃 최소 48×48px. 핵심 컨트롤(재생/녹음/추가)은 화면 하단 1/3.
2. 모든 편집 동작은 실행취소 가능(스냅샷 기반 undo 스택 20단계).
   삭제 등 파괴적 동작은 확인을 받는다.
3. 로딩·처리 중에는 반드시 시각적 진행 표시를 보여준다.
4. 레이블은 쉬운 한국어. 예: Quantize→"박자 맞추기", Gain→"소리 크기",
   Pan→"좌우 위치". 버튼 문구는 동사형("저장하기"), 흐름 내내 같은 단어 유지.
5. 세로 모드 우선. 빈 화면에는 항상 일러스트+다음 행동 안내(CTA)를 넣는다.

## 5. 바이브코딩 운영 수칙

### 5.1 피드백 프롬프트 공식

AI에게 "버그"를 말할 때는 항상 5요소로:

```
[화면/기능] 타임라인에서
[기대] 클립을 드래그하면 1/4마디 단위로 붙어야 하는데
[실제] 스냅 없이 자유롭게 움직이고 손을 떼면 원위치로 돌아감
[재현] 트랙 2의 두 번째 클립을 오른쪽으로 드래그 → 100% 재현
[환경] iPhone 13 / iOS Safari, 콘솔 오류: (있으면 붙여넣기)
원인을 먼저 설명하고, 수정 범위를 말한 뒤 고쳐줘.
```

- 나쁜 예: "드래그가 이상해요 고쳐줘" → AI가 추측으로 엉뚱한 곳을 수정
- "그리고"가 3개 이상 들어가는 요청은 쪼개세요. **1프롬프트 = 1관심사**

### 5.2 버전 관리 습관

- Phase 완료마다: `git add -A && git commit -m "phase-N: 요약"` (+ 태그 권장)
- AI가 크게 망쳤을 때: 고치는 프롬프트를 3번 반복하지 말고 `git reset --hard`로 마지막 커밋 복귀 → 프롬프트를 더 구체화해서 재시도
- 컨텍스트가 길어져 AI가 헤매면: **"현재 아키텍처를 5줄로 요약하고, projectStore의 실제 타입 정의를 보여줘"** 로 리프레시 후 계속

### 5.3 AI 폭주 방지

- 요청 밖 기능을 만들었으면: "요청 범위 밖 변경은 되돌리고, 변경 파일 목록을 다시 보고해줘"
- 라이브러리를 마음대로 추가했으면: "새 의존성 추가는 사전 승인 규칙 위반. 제거하거나 필요성을 설명해줘"
- 거대한 리팩터링을 제안하면: 먼저 "변경 범위와 위험을 5줄로" 요구 후 승인

### 5.4 오디오 앱 특유의 함정 (미리 알고 시작하기)

1. 브라우저 자동재생 정책 — 소리는 반드시 사용자 탭 이후 (Phase 1에서 구조로 해결)
2. iOS 무음 스위치 — 코드로 감지 불가, 안내 문구로 대응 (Phase 10)
3. 블루투스 이어폰 — 개발 중 타이밍 판단 금지, 유선으로
4. `setTimeout` 오디오 스케줄링 — 절대 금지, Transport만 사용
5. 파형 렌더링 — 매 프레임 재계산하면 스크롤이 죽음, 반드시 피크 캐시
6. 기기별 샘플레이트 차이(44.1k/48k) — 리샘플은 브라우저에 맡기고 가정하지 말 것

## 부록 B. 자주 만나는 오디오 버그 → 디버깅 프롬프트

| 증상 | 붙여넣을 프롬프트 |
|---|---|
| iOS에서 소리가 전혀 안 남 | "Tone.start()가 사용자 탭 핸들러의 동기 흐름에서 호출되는지 호출 경로를 추적하고, context.state를 주요 지점에서 로깅해줘. suspended면 다음 탭에서 resume하는 방어 코드를 확인해줘." |
| 박자가 점점 밀리거나 틱틱거림 | "오디오 타이밍에 setTimeout/setInterval을 쓰는 곳이 있는지 전수 검색하고 전부 Transport 스케줄로 바꿔줘. rAF UI 갱신과 오디오 스케줄이 분리돼 있는지도 확인해줘." |
| 두 번째 재생부터 소리가 겹침 | "정지/재생 반복 시 이전 스케줄이 정리되는지 확인해줘. Transport 이벤트와 Player가 stop 시 clear/dispose되는지 점검하고 누수를 고쳐줘." |
| 녹음 클립이 반 박자 밀림 | "녹음 시작 시점의 Transport 위치 기록 로직과 배치 계산을 보여주고, 고정 오프셋 보정값이 설정의 '녹음 타이밍 보정'과 연결되는지 확인해줘." |
| 타임라인 스크롤이 버벅임 | "타임라인 렌더를 프로파일해줘: 재생헤드 rAF가 클립 컴포넌트 리렌더를 유발하는지, 파형이 매 프레임 재계산되는지, 뷰포트 밖 클립이 렌더되는지 순서로 점검하고 고쳐줘." |
| 새로고침 후 프로젝트 깨짐 | "저장된 JSON과 현재 스키마 버전을 비교 로깅하고, 로드 시 검증+마이그레이션 단계를 추가해줘. 깨진 필드는 기본값으로 복구하고 사용자에게 알려줘." |
| 내보낸 파일이 무음 | "마스터 → MediaStreamAudioDestinationNode 연결 그래프를 그려서 보여주고, MediaRecorder가 start된 시점과 Transport 재생 시점의 순서를 점검해줘." |
